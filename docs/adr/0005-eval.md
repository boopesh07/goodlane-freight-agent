# ADR 0005 — Evaluation approach

**Status:** Accepted (revised — see "Revision" below)

## Context
The brief asks for a small eval set (≥1 core workflow), a score, and what we'd
improve. The first version of this eval was a single tier: a 3-case golden set
run against the free-query agent, graded by substring `must_mention`. That was
too weak to trust:

- **Substring matching gave false positives.** `"$"`, `"Goodlane"`, and
  `"Recommended"` appear in almost any answer, so a case could pass without
  being correct, and nothing caught a hallucinated fact.
- **One workflow, one load.** All three cases were the same load (29372343); no
  messy or negative paths (low-confidence match, compliance block, unknown
  carrier/load).
- **It only tested the LLM agent.** The actual core workflow is the
  *deterministic intake pipeline* (ADR 0006), and extraction quality (the messy
  voice data) was never measured — even though those are where regressions and
  errors actually live.
- **It needed an API key and was non-deterministic**, so it couldn't run in CI
  and the "3/3 = 100%" score wasn't very meaningful.

## Decision — three tiers
Evals are split by what they isolate, so each is reproducible where it can be
and gates the right surface.

### Tier 1 — deterministic intake pipeline (`eval/intake.golden.json`, `scripts/eval-intake.ts`)
Runs the **real** pipeline (`runEmailIntake` / `runCallIntake` — the same logic
`/api/intake` assembles) over labeled email + call records and asserts the exact
structured outputs the broker relies on: carrier + load resolution and
`matchedBy`, confidence and `needsHumanVerification`, compliance flags,
cross-reference validation, best offer, scope, and the recommendation. No LLM,
no API key — **fully reproducible, and gated in CI.** Cases deliberately cover
the happy path *and* the messy paths: compliance authority/insurance blocks,
low-confidence structured matches, fuzzy load-id recovery, carrier-name-only
resolution, cancelled loads, and cross-record best-rate retrieval.
Current score: **9/9**.

### Tier 2 — extraction field accuracy (`eval/extraction.golden.json`, `scripts/eval-extraction.ts`)
Hand-labeled ground truth (from the transcript text) for a sample of the messy
calls. Grades the committed extractions in `data/transcripts.json` and reports
**per-field accuracy** with normalization-aware comparisons (MC on digits,
company on fuzzy similarity, equipment on token containment). Deterministic and
CI-gated by default; `--live` re-runs the LLM extractor. It gates on an overall
threshold plus a per-field floor.
Current: **overall 96% (46/48)**, with a real, visible weakness —
**`load_reference` 60%**: the extractor garbles spoken multi-digit load numbers
(e.g. "twenty-nine million…" → a 9-digit id). The deterministic pipeline already
*recovers* from this via fuzzy id matching (proven in Tier 1), but the eval
surfaces it honestly rather than hiding it.

### Tier 3 — end-to-end agent (`eval/golden.json`, `scripts/eval.ts`)
Runs the free-query agent with live tools and grades three axes per case:
1. **text assertions** — `must_match` regexes **and** `must_not_match`
   (anti-hallucination — e.g. an unknown carrier/load answer must not contain a
   dollar figure).
2. **tool assertions** — `tools_used` were called (and optionally
   `tools_not_used` were not).
3. **LLM judge** — a separate model call grades *faithfulness to the retrieved
   tool results* and rubric adherence, gated at a minimum score.

Cases now include negative/honesty paths: confirm-before-quoting on a
low-confidence match, a compliance block, and an honest "no record" answer for a
non-existent carrier/load. Needs `OPENAI_API_KEY`; non-deterministic, so it runs
**locally / on demand**, not in PR CI.

## What I'd improve next
- Grow the Tier 1/2 labeled sets (more loads, more carriers; label MC suffix and
  availability fields in Tier 2).
- Fix the Tier 2 `load_reference` weakness at the source — a deterministic
  spoken-number parser for load ids in extraction, then re-measure.
- Track Tier 2 over time (precision/recall per field) and alert on regression.
- Run Tier 3 as a scheduled (nightly) job with the judge, and add an
  inter-judge agreement check on a small human-graded subset.

## Consequences
- CI gates the two deterministic tiers on every PR with no secrets.
- Each tier isolates a different failure mode (resolution/reasoning vs.
  extraction vs. agent behavior), so a red eval points at the right layer.

## Revision
Superseded the original single-tier substring eval. The legacy 3 agent cases are
retained and hardened inside Tier 3.
