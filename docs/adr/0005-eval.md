# ADR 0005 — Evaluation approach

**Status:** Accepted

## Context
The brief asks for a small eval set (≥1 core workflow), a score, and what we'd
improve. We want an eval that exercises the agent's retrieval + reasoning +
honesty and that survives changes to extraction.

## Decision
- **A golden set of agent workflows** (`eval/golden.json`) run by a harness
  (`scripts/eval.ts`) that calls the real agent (`generateText` with the live
  tools) and asserts two things per case:
  1. **`must_mention`** — required facts/ids appear in the answer.
  2. **`tools_used`** — the expected tools were actually called (captured via
     `onStepFinish`).
- **Three cases** covering distinct capabilities the brief lists:
  - `timeline-29372343` — multi-source timeline + recommended action.
  - `best-rate-29372343` — "best rate on offer" retrieval with sourcing.
  - `draft-reply-29372343` — drafting a carrier reply (asserts `draft_email`).
- The harness prints per-case pass/fail, tools called, a response preview, and an
  aggregate score. Current score: **3/3**.

## What I'd improve
- Add a compliance/availability case and a deliberately low-confidence load case
  to assert the human-verification path.
- Add an *extraction* eval (compare `extracted` call fields against a hand-labeled
  sample) — distinct from this agent-level eval.
- Move from substring `must_mention` to a small LLM-judge for semantic grading,
  and wire the eval into CI as a non-blocking nightly (it needs an API key).

## Consequences
- The eval measures end-to-end agent behavior, not just the extractor.
- It requires `OPENAI_API_KEY`, so it runs locally / on demand, not in PR CI.
